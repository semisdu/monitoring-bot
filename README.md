# 🤖 Monitoring Bot

Универсальная система мониторинга инфраструктуры с поддержкой Docker, PVE, PBS и мультиязычностью.

## 📋 Возможности

- ✅ Мониторинг Docker контейнеров (алерты при падении)
- ✅ Мониторинг логов внутри контейнеров (ошибки, exceptions)
- ✅ Мониторинг Proxmox VE (статус VM, CPU, RAM)
- ✅ Мониторинг бэкапов PBS (возраст, свободное место)
- ✅ Проверка доступности сайтов (HTTP/HTTPS)
- ✅ Ежедневные отчёты в Telegram
- ✅ Мультиязычность (🇷🇺 Русский, 🇺🇦 Українська, 🇬🇧 English)
- ✅ Универсальная архитектура - добавляй любые серверы без изменения кода

## 📖 История создания

### 🗓️ 2 января 2026 года

К тому моменту мы уже месяц тестировали один образовательный сайт. 
Каждое утро происходило одно и то же: сайт ложился в хлам.

### 🌙 Ночной кошмар

А причина была проста:
- Ночью PBS (Proxmox Backup Server) начинал свою работу
- Бэкапил все виртуальные машины
- Сайт на VM притормаживал, подвисал, а иногда и падал
- К утру мы получали упавшую базу и кучу вопросов

### 💡 Момент прозрения

Я сидел удалённо, смотрел на очередной упавший сайт и понял:
> "Я не могу каждый раз вручную проверять, что пошло не так. 
> Нужен инструмент, который будет делать это за меня".

### 🔨 Как рос проект

Сначала были простые статические скрипты.
Потом — первые кнопки в Telegram, которые просто отправляли команды.

Но чем больше становилось серверов и контейнеров, тем сильнее росло понимание:
> Статические кнопки — это тупик.

Так родилась идея **динамического меню**, которое само подстраивается под то, что реально есть в системе:
- Добавил новый сервер → кнопка появилась сама
- Убрал контейнер → исчезла из меню
- Не нужно переписывать код, просто правлю конфиг

## 🔑 Настройка SSH-доступа

Для мониторинга серверов боту нужен доступ по SSH. Самый безопасный способ — использовать SSH-ключи.

### 1. Создайте отдельную пару ключей для мониторинга (на вашем локальном компьютере)
Не используйте свой личный ключ! Создайте новый, например, с именем `id_ed12345`:

ssh-keygen -t ed12345 -f ~/.ssh/id_ed12345 -C "monitoring-bot-key"

*   `-t ed12345` — тип ключа (рекомендуется).
*   `-f` — путь и имя файла для сохранения ключа.
*   `-C` — комментарий (можно указать название проекта).

Вы можете задать пароль (passphrase) для ключа, но бот не сможет его ввести автоматически, поэтому для работы в фоне пароль лучше оставить пустым (просто нажать Enter).

После выполнения у вас появятся два файла:
*   `~/.ssh/id_ed12345` — **приватный ключ (НИКОМУ НЕ ПОКАЗЫВАЙТЕ ЕГО!)**
*   `~/.ssh/id_ed12345.pub` — публичный ключ (его можно копировать на серверы).

### 2. Скопируйте публичный ключ на ваш сервер
Выполните эту команду **для КАЖДОГО сервера**, который хотите мониторить. Замените `your_username` на имя пользователя на сервере (например, `root` или `your_username`) и `192.168.1.100` на IP-адрес или домен сервера.

ssh-copy-id -i ~/.ssh/id_ed12345.pub your_username@192.168.1.100

Вас попросят ввести пароль пользователя для этого сервера. После успешного выполнения ключ будет добавлен в файл `~/.ssh/authorized_keys` на удаленном сервере, и в следующий раз пароль не потребуется.

Если команда `ssh-copy-id` недоступна, скопируйте ключ вручную:

cat ~/.ssh/id_ed12345.pub | ssh your_username@192.168.1.100 "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"

### 3. Укажите путь к ключу в конфиге бота
В вашем файле `config/config.yml` для каждого сервера укажите имя файла вашего **приватного** ключа в параметре `ssh_key`:

servers:
  - id: "my-server"
    # ... другие параметры ...
    user: "your_username"  # Имя пользователя на сервере
    ssh_key: "id_ed12345"  # Имя файла приватного ключа в папке .ssh

### 4. Проверьте подключение
Убедитесь, что бот сможет подключиться:

ssh -i ~/.ssh/id_ed12345 your_username@192.168.1.100

Если вы зашли без запроса пароля — всё готово к запуску бота! 🚀
